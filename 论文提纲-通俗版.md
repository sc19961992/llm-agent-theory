# 从Transformer数学结构到Agent工程约束

> 本文论证：LLM Agent 的若干已知限制——任务分解的脆弱性、记忆缺乏时间动态、全自动闭环不可达——
> 不是工程上的暂时问题，而是 Transformer 架构的数学结构决定的必然结果。以下先陈述结论及其工程含义，
> 再逐环追溯推理过程。


## 结论

以下五条是本文推导的终点。先列出来，是因为每条都对应已有的工程实践经验——读者可能在日常工作中已经在遵守它们，但不一定清楚它们为什么必须成立。

**1. 提示词的信息结构精度决定输出可靠性的上限。** 模糊输入更容易落在模型收敛空间的覆盖范围之外，而在范围之外的输出不受训练收敛的保证。关键在于信息结构的歧义密度，而非 token 数量。Chain-of-Thought 增加了 token 数但减少了歧义——它在这个意义上是"更精确"的。

**2. 上下文驱动的分解决策不可信。** 模型在上下文窗口内做出的结构决策——任务如何切割、接口如何定义、模块如何划分——不受收敛保证。这不是经验观察（"有时会出错"），而是从前提推导出的结构结论。

**3. 关键分叉点需要外部仲裁。** 任务分解中，父节点的错误会传导至所有后代节点且子节点无法自行纠正。划分节点的决策本身依赖上下文窗口，而上下文不受收敛保证。因此，不可逆的高影响决策必须由系统外部介入——人或独立于模型的硬约束程序。

**4. 每个任务单元必须有可自动验证的完成标准。** Agent 的行为不由训练收敛（梯度下降）驱动，而由测试收敛（生成→验证→修正循环）驱动。测试收敛的收敛性取决于验证信号的覆盖度。没有自动可执行的验证标准，就没有收敛机制。

**5. 全自动闭环在原则上不可闭合。** 层级验证链向上追溯，终点是"此任务是否满足用户真实需求"。该判断的语义落在模型收敛空间之外——因为具体用户在具体时刻的具体意图不在训练数据的语义分布中，因而不在米田嵌入的截断投影范围内。根需求必须外部提供。


## 推理链

以下从表层工程现象开始，逐环追问"为什么"，最终落到 Transformer 的数学结构。

### 1. 为什么 Agent 必须依赖可验证目标？

模型训练完成后参数冻结。Agent 运行时不能修改权重——它是一种在上下文窗口内、通过"生成输出→检验→修正→再检验"的循环来逼近目标的过程。这个过程的有效性完全来自验证信号的覆盖度和精度。训练收敛（梯度下降）有损失函数的梯度作为导向信号；这种在上下文中的迭代没有等价的数学保证。当验证信号遗漏了某个隐含约束时，Agent 可以收敛到一个通过所有显式测试但不满足实际需求的状态——这不是能力不足，而是验证信号的信息量不足以唯一确定目标。

### 2. 为什么验证不能仅靠模型自查？

模型自查（self-verification）也是上下文窗口中的一次推理。上下文中的信息不受收敛保证——解释见下一环。

### 3. 为什么上下文中的信息不受收敛保证？

训练收敛的过程是：(a) 从训练数据的语义分布中筛选出有统计结构的关系，(b) 将这些关系压缩进参数空间中的低维表示。上下文窗口中的信息绕过了这个过程——它是每次推理时新注入的顺序化信息串，未经参数空间的结构筛选。

训练收敛保证的是"模型存储了什么"。Agent 依赖的是"模型在这个特定上下文中如何使用它存储的东西"。前者有收敛保证，后者没有。两者的认知性质不同。

### 4. 为什么任务切块能缓解——但又不能完全解决？

上下文窗口有有效利用边界：模型实际上能可靠利用的上下文长度远小于 token 上限（Lost in the Middle 效应）。单个任务的信息量在该边界内时，其语义被训练收敛覆盖的概率较高，测试收敛有条件运作。当任务信息量超出有效边界，部分约束必然落在收敛覆盖之外，测试收敛在盲区中失效。

任务切块将每个子任务的信息量缩小到有效边界内——不是优化性能，而是恢复测试收敛的工作条件。

### 5. 为什么"怎么切"不能交给模型？

任务分解需要理解全局结构：哪些子任务是耦合的、接口划在哪里、依赖关系如何。这些判断需要将全部相关信息同时装入上下文窗口才能做出。但如果任务本身的信息量已经超出了上下文有效边界（否则不需要分解），则分解决策的信息需求同样超出了该边界。

这是一个自指问题：判断一个任务是否需要分解，以及如何分解，这个判断本身的信息量可能已经超出了上下文窗口的有效承载量。在这种条件下，分解决策不受收敛保证是信息论上的必然，而非经验上的偶然。

### 6. 为什么根需求必须在系统外部？

层级任务分解形成验证链：子任务输出 → 父级验证 → 更上层验证 → … → 最终是否满足用户需求。每一层验证在各自的上下文窗口内进行，只持有局部信息。最顶层的判断——"此结果是否符合用户的真实意图"——其语义不在模型收敛空间内。

模型收敛空间覆盖的是训练数据的语义分布。具体用户在具体时刻的具体意图，在训练数据中没有标注，不在米田嵌入中，不在截断投影的覆盖范围内。这不是模型规模或训练数据量的问题——收敛空间的结构边界已决定了这一点。根需求来自收敛空间外部，因此全自动闭环在原则上不可闭合。

### 7. 为什么存在收敛空间的结构边界？

Transformer 注意力的数学形式（softmax(QKᵀ/√d)V）与 Nadaraya-Watson 核估计同构，在高斯过程先验下等价于后验均值计算——即在"语义空间"中做贝叶斯推断。这个语义空间由米田嵌入定义：一个概念由其与所有其他概念的全部关系所确定，在数学上是无穷维的。

但 Transformer 的参数空间的有效维度（intrinsic dimension）仅有数百到数千维——这是 Li et al. (2018)、Aghajanyan et al. (2020) 以及 LoRA 低秩有效性共同支持的经验事实。

模型学到的不是完整语义空间，而是它的一个低维截断投影。训练收敛的所有保证仅在该投影的覆盖范围内成立。超出此范围的语义关系，模型没有结构保持的映射。

Agent 面临的任务——尤其是开放式的、需跨领域组合推理的、依赖用户具体意图来定义"完成"的任务——本质上在反复触及这个边界。


## 推理链总图

```
注意力的数学形式
  → 米田语义空间中的贝叶斯推断
  → 参数维度有限 → 截断投影
  → 训练收敛仅覆盖截断投影内的态射
  → 上下文窗口中的信息未经过训练收敛筛选 → 不受收敛保证
  → Agent依赖测试收敛（上下文中的生成→验证→修正循环）→ 无数学保证
  → 上下文驱动的分解决策不受收敛保证 → 关键分叉点必须外部介入
  → 验证链上溯 → 根需求在收敛空间之外 → 全自动不可闭合
```

这条链上任意一步不成立，上述结论就需要修正。


## 工程含义

给定以上约束，Agent 工程设计的可行域如下：

- 将任务切至上下文有效边界内，使测试收敛有条件运作
- 每个单元配备可自动执行的验证标准
- 结构决策节点（任务划分、接口定义、需求消歧）由外部仲裁介入
- 接受根需求的外部性——人机协作在此结构下不是过渡方案

超出这个可行域的设计——依赖模型自行规划长程任务、以模糊需求驱动多步自主决策、试图在现有架构内实现闭环自举——面临的不是工程上的尚未解决，而是当前数学结构下的不可解决。
