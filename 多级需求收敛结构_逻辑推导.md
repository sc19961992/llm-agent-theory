## 推导一：任务必须分解

| 步骤 | 依赖 |
|------|:--:|
| 多数有意义的任务超出单次文字输入边界 | B1 |
| 超出边界的问题不受收敛约束覆盖，无法单次完成 | P1 + P2 |
| → 必须分解为落入边界内的子任务 | P1 + P2 |

## 推导二：子任务须可验证，分解形成层级

| 步骤 | 依赖 |
|------|:--:|
| 子任务的目标须在文字输入边界内设定 | P1 |
| 子任务需要可验证目标方可触发测试收敛 | P3 + P4 |
| 分解本身是任务的一个实例——受同一边界约束 | P1（逻辑实例化） |
| 分解过程本身需在上下文窗口内完成，需测试收敛 | P1 + P3 |
| → 层级形成：上层收敛产出下层目标 | P3 + P4 |

## 推导三：层级节点须硬约束

| 步骤 | 依赖 |

## 结论：多级需求收敛结构

> 模型的能力边界由文字输入边界划定。现实任务通常超出该边界，必须分解为落入边界内的子任务方可触发测试收敛。分解本身是一次收敛过程——在上下文窗口内为子任务设定可验证目标——由此自然产生层级：上级收敛的产出是下级收敛的目标。
>
> **硬节点约束。** 上下文未经收敛约束。流程节点的设定是一次性向下传导——节点错误使下层全部子任务目标偏离，代价乘性。故层级之间的关键分叉点须由人工定义或程序硬约束固定，二者均经独立验证，不依赖上下文的未收敛映射。
>
> **复杂度指数增益。** 每向下展开一层，子任务数量倍增，每个子任务各自执行测试收敛。层级加深带来覆盖面的指数扩张——这是分解的必然结构效应，不是可选策略。代价是高层分解本身的收敛成本，该成本同样受上下文窗口限制，需要在制定阶段收敛。
>
> **根需求的人工锚定。** 测试收敛依赖可验证目标，验证标准递归上溯。最高层目标——用户真正需要什么——其验证权威不可由系统自举。多级需求结构在此终止。全自动闭环收敛在根节点处原则上不可闭合。
